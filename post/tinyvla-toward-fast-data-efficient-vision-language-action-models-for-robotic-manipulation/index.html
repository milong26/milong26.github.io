论文#

实验#

  1. 模拟基准测试：我们在 MetaWorld 上评估我们的方法。MetaWorld [36] 中的 50 个任务可以分为多个级别 [37]，即简单、中等、困难和非常困难。
  2. 真实的机器人设置：TinyVLA 既在使用使用 Franka Panda 7Dof 机器人手臂的单臂设置上进行评估，也可以在使用两个 UR5 机械臂的双手动设置上进行评估，如图 3 所示。单臂场景通过固定在机器人两侧的两个外部 ZED 2 立体摄像头感知。双手动机器人的场景由手腕上的两个摄像头捕捉，顶部还有一个额外的摄像头。这些相机是 Realsense D435i。
  3. 任务：在单臂设置中，有五个任务：1） 关闭抽屉 （CloseDrawer），2） 将粉红色立方体堆叠在蓝色立方体的顶部 （StackCubes），3） 打开盒子的盖子 （OpenBox），4） 将网球放入球盒，以及 5） 直立翻倒的杯子 （FlipMug）。在双手动人实验中，我们设置了三个涉及两只手臂合作的任务：1） 将面包转移到盘子上 （TransferBread），2） 拉开袋子的拉链并将网球放入其中 （PlaceTennisBag），以及 3） 将立方体堆叠在盘子上 （StackCubes）。值得注意的是，任务的动作空间差异很大。例如，翻转杯需要机器人进行大范围的旋转才能将夹持器横向插入杯中，这与堆垛立方体完全不同，后者是拾取和放置型的。跨度同一任务中不同轨迹的长度也明显不同，例如，堆栈立方体轨迹的长度范围为 100 到 300。这为 TinyVLA 在学习执行这些任务方面带来了更多挑战。
  4. 数据收集：我们通过远程作收集数据集。我们记录来自两个摄像头视图的 RGB 流和机器人状态，例如整个机器人控制过程中的关节位置。我们将机器人夹持器的宽度记录为 0 到 1 之间的值，其中 0 表示完全关闭，1 表示完全打开。TinyVLA 预测 7 维动作，包括位置 （x， y， z）、旋转 （roll, pitch, yaw） 和 （gripper_width）。对于所有任务，除了 remove the lid of the box 任务外，我们不会添加额外的干扰项，以便更好地评估模型对干扰项的泛化能力。我们总共为每个任务收集了 100 个轨迹，以平衡所有 5 个任务之间的数据分布。

可以看出来实验的train data需要自己收集，包括RGB 流和机器人状态，输出预测的7维动作

freejoint的qpos一共有七个数，其中，qpos[02]是freejoint自身坐标系原点在世界坐标系的位置，qpos[36]是自身坐标系相对于世界坐标系的四元数；qvel一共有六个数，其中，qvel[02]是freejoint速度在世界坐标系下的表示，而qvel[35]是freejoint角速度在自身坐标系下的表示

tinyvla用论文结构、内容上解读+代码复现

期刊的，好一点，参考这个结构 2024 IEEE Robotics and Automation Letters 学习它的结构的话，就看它方法怎么写的和实验怎么组织的

怎么写

  1. 摘要：现状、问题、本文提出xxx、两个优势对应前面的问题、框架包含两个部分、再仿真和真实测试、实验结果表明最基础的功能、此外还能泛化、最终的结论
  2. 介绍
    1. 机器人操作的过去和现在（标题是for robotic manipulation）：描述问题，为什么复杂、传统、现代方法
    2. VLA的发展和瓶颈：vla模型、列举几个模型、缺陷描述和原因、缺陷对实际的影响，提出问题
    3. 本文方法设计：发现的理论也就是后面改进的基础、怎么做的、实验结果表明（不用写数据）
    4. 实验结果对比：模拟还是仿真结果都好，比如具体的（可以写数据）
    5. 贡献三个方面：提出新的架构、实验验证、证明xxx
  3. 相关工作（为什么是这三个？是根据前面来的还是根据后面来的？说是研究问题的三个抽象层级，从通用到具体，应该是来自method的技术拆解，看到后面的时候回来看一下）相关工作写的时候一句话：别人已经做了A/B/C，但存在问题D/E，我们在这些维度改进。可以从领域定义、已有方法分类、局限总结、本文的切入点来写
    1. 基础模型——vlm：功能和影响、推理成本高的问题、一些轻量化研究
    2. 应用范式——用于机器人学习的vlm：？
    3. 任务与训练——多任务机器人学习：主流方法列举，最后本文提出xxx

看到这里缓一下，总结前面三段的写作方式

  1. 摘要：问题-方法-效果
    1. 【现状+问题】+方法描述+方法结构+结果（实验说明、结果对比、附加能力）+最终总结
    2. 结果需要写一点数据对比
    3. 整体篇幅大概每个点一句话，不展开推理
  2. 引言：论证“为什么这个研究不得不做”以及“为什么我的方案是合理且有效的”
    1. 现状：任务背景，为什么难
    2. 发展和问题：方法简述+引出核心问题
    3. 我的工作：提出xx，设计点是什么，解决了什么问题
    4. 实验结果：强调趋势结论，只做“可信性预告”，不做总结
    5. 贡献总结：可以写提出xx、实验结果（不能重复表达方式）、结论
  3. 相关工作：别人做了xx，但存在问题xx，我们在这些维度切入
    1. 先将method拆分领域
    2. 然后再写这个领域的问题难点、已有方法列举、局限总结、本文切入点

  1. 方法
    1. 这一章的开头，说明设计理念，放示意图
    2. 然后就解释方法
  2. 实验
    1. 先说实验研究的问题
    2. 实验设置

然后在内容上对tinyvla做一个总结：

  • 方法部分
    • vlm
      • 基于Pythia语言模型基座和LLaVA的训练流程，专门预训练了一个更紧凑的视觉语言模型家族（参数量从70M到1.4B，分为S、B、H三个版本）
      • 微调方法：lora+”为应对输入序列长度的不确定性，我们采用了自适应池化层和层归一化处理，从而得到结构固定、尺寸紧凑的特征表示。“这个是啥操作？应该不复杂吧
    • 动作输出头
  • 实验部分
    • 仿真环境是metaworld
    • baseline选择diffusion policy，multimodal diffusion openvla
    • 居然还设计了一个消融实验：通常，消融实验的目的是通过“控制变量法”（比如去掉某个模块，或替换某个模块）来证明模型中各个组件设计的必要性。本文在消融实验部分引入“失败案例分析（Failure case analysis）”是非常巧妙的写法 1。在探究“VLM模型大小与性能的权衡”时，作者不仅用定量数据展示了模型越大成功率越高，还通过定性的失败归因（例如：0.4B模型失败是因为指令理解错误，而增加到1.3B模型后解决了定位和理解问题）来深入解释了为什么较小的模型效果不好 1。这使得消融实验不仅停留在数字对比，更提供了具有说服力的机制解释。

“We identify two key factors in existing VLA models that contribute to their high inference latency: (1) they are built on large vision-language models, often exceeding 7 billion parameters, and (2) they generate discrete action tokens autoregressively, requiring repetitive inference for each degree of freedom.“这一段方法里面没有验证，难道是实验部分才有？引言中的这句话确实是引出本文方法的“核心动机（Motivation）”，但它并不只是一个纯理论假设，作者在消融实验的 V.C 小节（Which Part of TinyVLA Makes it Fast?）结合 Table IV 进行了严谨的控制变量证明 5。

流程摘要（简短）：

- train_tinyvla.main
    - 调用 load_data(...) 创建 train_dataset (EpisodicDataset) 和 llava_pythia_process (LlavaPythiaProcess)。
- EpisodicDataset.**getitem**
    - 从 hdf5 读出单帧每个 camera 的 raw image (H,W,3)。
    - 如果 self.imsize != 原图宽，做 cv2.resize(image, (320, 180))（注意 cv2.resize 的参数是 (width, height)）。
    - 将每个 camera 的图像 np.stack -> torch.Tensor，BGR->RGB（如果包含 'top'），再 einsum -> (k, c, h, w)。
    - 如果 policy_class 包含 'diffusion'，self.augment_images = True，初始化 self.transformations，然后执行：  
        for transform in self.transformations:  
        image_data = transform(image_data)  
        （这里把 torchvision.transforms 直接作用在一个 4D tensor (k,c,h,w) 上）
    - image_data /= 255，构造 sample 并交给 llava_pythia_process.forward_process(sample)。
- LlavaPythiaProcess.forward_process / parse_image
    - parse_image 将 (k,c,h,w) -> permute -> (k,h,w,c).numpy()，并交给 self.processor.preprocess(...)（transformers 的 CLIPImageProcessor / SiglipImageProcessor）。
    - processor 返回 pixel_values (B, C, H2, W2)，这些 H2,W2 是 processor 或前面 transforms 最终产生的大小（在你的 run 中是 336×597）。
    - 后续把 image chunk 给 model -> 导致 CLIP 的 position embedding mismatch。

哪里导致 shape 异常（结论）：

- 两处会改变尺寸：cv2.resize(..., (320,180))（把原 480×480 -> 180×320），以及 Llava 的 image processor（可能会 resize 到模型预期）。
- 但你看到的异常尺寸 336×597 主要来源于在 EpisodicDataset 中对整批 (k,c,h,w) 应用 torchvision.transforms 的那一段。torchvision 的图像变换通常作用于单张 (C,H,W) 或 PIL.Image，而把它直接作用在 4D tensor 上会产生未定义/意外的行为（导致大小被错误放大/重排），最后传入 processor 得到奇怪的 H×W（336×597）。
