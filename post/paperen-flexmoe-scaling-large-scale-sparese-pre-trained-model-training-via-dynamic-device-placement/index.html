标题：FlexMoE:度量 大规模 稀疏 预训练模型训练 通过 动态设备放置 一句话：改进moe

abstract#

背景#

趋势：用大规模的(large-scale)预训练模型 but：它用了很多密集的代数，需要硬件资源

目前：稀疏门控专家混合(sparsely-gated Mixture-of-Experts,MoEs)参考 but：因为路由不平衡(route imbalance)和波动(fluctuation)问题，这种稀疏条件计算在实际系统中可能不如预期的有效。

  • 这两种都是MoEs通讯上的问题吗？
  • 专家到底是什么

这篇#

提出的算法#

提出：FLEXMOE 分类：DNN训练框架 优点：系统地、透明地解决了动态递归造成的效率低下问题

工作步骤#

  1. 分析了之前的MoE模型 促使作者想办法克服路由不平衡(route imbalance)和波动(fluctuation)问题
  2. 在DNN上引入了一个新的调度模块 干啥：监控数据流，制定调度计划，并根据实时数据流量动态调整模型到硬件的映射
  3. 启发式算法 在训练过程中动态优化设备放置
  4. 在NLP、视觉模型上进行了实验 结果：与现有系统相比，flexmoe比deepspeed平均好1.7倍，最高2.1；比fastermoe平均好1.3倍，最高1.45

介绍#

前提#

大规模预训练模型PTMs很有用，但是要模型质量好就要参数多，训练起来非常昂贵。 改进：利用MoE结构，现在用处很多了 但是：MoE结构也有缺点，需要很多数据，还很消耗计算资源 需要：解决隐藏在现有系统中的数据管理问题，并进行优化以克服这些挑战。

分布：以前的分布式不够，因为单个MoE层能轻易超过GPU上限内存 改进：专家并行(expert parallelism)将专家分配到多个GPU中，以适应有限的GPU内存。 具体：对每个训练token，在门网络确定好路线之后，它们（每个训练token）会被发送到GPU那去，什么GPU？有它们目标专家在的GPU

工作量不均衡#

来源：MoE层稀疏、且有条件的计算性质带来的 具体：输入的token被组织在一个细粒度的水平，并通过门控网络路由到个人的专家。

  • 组织是什么意思
  • 细粒度是什么水平
  • 这个是它稀疏、有条件的计算性质的具体体现？ 表现：输入token显示出对专家的高度不均匀偏好，并且路由结果在训练过程中不断变化。 结果：这种工作量不平衡可能导致效率显著下降和严重的资源利用不足。

目前的景观(landscape??)#

  • 为了解决工作量不均衡的问题: 在每个MoE层，利用一个辅助平衡损失函数来促进平衡路由(分配)
  • 为了控制系统效率（工作负载平衡）和统计效率（模型质量）之间的权衡: 平衡损失还与可调惩罚权重相关联
